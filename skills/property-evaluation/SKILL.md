# 房产评估数据提纯

把原始房产记录提纯为可用于评估的民用住宅小区清单。核心原则是：先核验名称和用途，再按人工指定城市筛选，最后按附件一格式输出；不要擅自改变用户没有要求改变的估价算法。

## 执行流程

1. 确认输入文件和目标格式。

   - 找到用户提供的 CSV/XLSX，以及作为“附件一”的示例或上一版汇总表。
   - 先读取表头、行数和样例值，确认面积、单价、总值、项目名称、小区名称、法定用途、城市等字段的实际列名。
   - 将输出第一张表固定为附件一的字段顺序：`city`、`面积㎡`、`评估单价`、`评估总值`、`小区名称`、`备注`。保留附件一已有的基本格式。

2. 确认城市。

   - 城市由人工指定；如果用户已明确城市，严格只保留该城市。
   - 用户未指定时，以深圳为默认城市，并在开始处理时告知“未指定城市，按深圳处理”。
   - 如果文件中有多个城市且用户的意图不明确，先询问目标城市，不要把多城市混在一张结果表中。

3. 读取元数据并标准化小区名称。

   - 将附件一来源数据与元数据按原始项目/房屋记录对应起来；不要只按字符串完全相等匹配，要结合地址、栋座、房号等字段确认对应关系。
   - 对每个候选住宅小区，用联网搜索核验“原始名称来自哪个正式小区名称”。优先参考政府/开发商/物业/地图/房产平台等能显示项目正式名称的页面，并结合城市和地址避免同名误配。
   - 记录标准化名称；名称变更必须可解释，例如去掉“花园、社区、一期”等冗余后缀，或将网上常用名与登记名统一。无法确定时保留原名，并在备注写明“待人工确认”，不要编造名称。
   - 名称中出现“大厦”不等于一定是非住宅；必须根据联网资料判断其是否为住宅小区。

4. 剔除非民用住宅。

   - 优先依据“法定用途”字段筛选。剔除 `business_apartment`、`bachelor_apartment`、公寓、商务公寓、商业、办公、公司、厂房、车间、工业、宿舍、仓库、酒店及其他非住宅用途。
   - 即使法定用途写作 `residence`，如果联网资料明确显示项目属于商业办公/公寓混合项目，仍然剔除，并在“筛选说明”写清依据。
   - 不要仅凭项目名称中的“大厦”“广场”“中心”等词剔除；先联网核验。
   - 同一住宅小区的多套房源应保留为多条评估记录，不要误去重。只有重复抓取的同一条记录才去重，去重依据应优先使用原始行号、房屋地址或其他唯一字段。

5. 过滤城市并准备数值。

   - 只保留目标城市的民用住宅记录；例如默认目标城市为深圳时，南昌等其他城市全部剔除。
   - 保留来源数据的面积、评估单价和评估总值。若总值缺失且附件一的现有规则明确要求计算，才按“面积 × 评估单价”补算并沿用原有舍入方式。
   - 不要自行改用均值、中位数、去极值、加减 1 或其他估价算法。只有用户明确提出时，才修改数值算法，并先说明影响范围。

6. 生成工作簿。

   - 第一张表命名为 `房产评估汇总表`，按附件一字段和顺序输出，仅包含目标城市、民用住宅记录。
   - 第二张表命名为 `筛选说明`，至少包含：原 CSV 行号、项目名称、原小区名称、法定用途、标准化小区名、保留/剔除结果、处理说明。
   - “筛选说明”保留所有原始记录，既说明保留项，也说明每条剔除项的原因，尤其标记非目标城市记录、非民用用途和名称待确认项。
   - 默认另存新文件，不覆盖用户已有文件。文件名使用原文件名加 `_仅{城市}`，例如 `房产评估汇总表_生成3_仅深圳.xlsx`。

## 质量检查

生成后必须检查：

- 汇总表只出现目标城市，且行数与“筛选说明”中目标城市且保留的记录一致。
- 汇总表没有公寓、商业办公、公司、厂房、车间、工业、宿舍等非民用住宅项目。
- 每条汇总记录都能在元数据中找到对应原始行号或可解释的匹配依据。
- 标准化名称不确定的记录已写入备注，不得静默猜测。
- 面积、单价、总值字段没有因筛选而错位；总值如有补算，应可由面积和单价复核。
- 工作簿能正常打开，两个工作表名称和附件一字段顺序正确。

向用户交付时，简要说明目标城市、保留/剔除数量、主要剔除类别，并提供生成文件的绝对路径链接。
