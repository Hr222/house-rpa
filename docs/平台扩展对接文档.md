# 平台对接指南

本文档用于后续接入第二个平台或更多平台时参考。当前项目已经按多平台结构拆分，新平台按本文步骤接入即可。

## 1. 对接目标

新增一个平台时，需要解决四件事：

1. 打开该平台的常驻首页。
2. 支持人工登录后确认就绪。
3. 接收询价请求并完成采集。
4. 把结果转换成统一的 `PlatformResult`，交给服务层继续计算最终值。

## 2. 现有扩展点

当前项目已经预留的插槽如下：

- `app/platforms/base.py`
  统一平台适配器接口。
- `app/platforms/ke.py`
  贝壳平台适配器，可作为新平台模板。
- `app/platforms/ke_constants.py`
  平台固有常量的放置方式示例。
- `app/ke_adapter.py`
  平台真实采集逻辑示例。
- `app/registry.py`
  平台注册入口。
- `app/runtime.py`
  负责平台会话、保活、任务执行。
- `app/service.py`
  汇总平台结果并计算 `quoteAvg`、`dealAvg`、`finalPrice`。

说明：

- 业务目标是面向“当前所有已接入平台”。
- 但当前实现仍是先取第一个 `SUCCESS` 平台结果。
- 因为当前只接入贝壳，所以现阶段结果上没有差异。

## 3. 新平台建议文件结构

以平台代号 `ajk` 为例，建议新增这些文件：

```text
app/
├─ ajk_adapter.py
└─ platforms/
   ├─ ajk.py
   └─ ajk_constants.py
```

职责划分：

- `app/platforms/ajk.py`
  平台适配器，负责接入统一接口。
- `app/platforms/ajk_constants.py`
  平台固有常量，例如首页地址、面积档位、固定选择器。
- `app/ajk_adapter.py`
  真实采集逻辑，负责页面交互、点击、翻页、抓取、详情页处理。

## 4. 必须实现的接口

新平台适配器必须继承 `PlatformAdapter`，并实现以下方法。

### `open_session`

作用：

- 打开平台首页。
- 返回 `PlatformSession`。

要求：

- `code` 必须唯一，例如 `ke`、`ajk`、`lianjia`。
- `name` 为日志和展示使用的平台名称。
- `start_url` 为该平台的常驻首页。
- `page` 为 nodriver 当前页面对象。

### `collect`

作用：

- 执行一次完整的单平台询价采集。

输入：

- `browser`
- `session`
- `request`

输出：

- `PlatformResult`

### `check_ready`

作用：

- 检查当前平台是否已登录且能接单。

返回：

- `(True, "READY")`
- 或 `(False, "原因描述")`

### `keepalive`

作用：

- 执行轻量保活。

默认可以直接复用 `check_ready`。如果平台必须通过刷新、回首页、滚动等方式保活，可以在平台适配器里覆盖这个方法。

## 5. `PlatformResult` 要求

平台采集最终必须返回统一结构：

- `name`
- `status`
- `community_avg_price`
- `quote_prices`
- `deal_prices`
- `reason`
- `request_id`
- `detail_url`
- `elapsed_seconds`
- `listing_snapshots`

字段含义：

- `community_avg_price`
  平台详情页提供的小区参考均价。当前服务层会优先把它当作 `quoteAvg` 候选。
- `quote_prices`
  在售单价列表。
- `deal_prices`
  成交单价列表，注意这里应该是面积筛选后的结果。
- `listing_snapshots`
  用于日志打印和演示展示的房源摘要。

`status` 当前建议使用以下值：

- `SUCCESS`
- `NO_DATA`
- `WAIT_MANUAL_VERIFY`
- `LOGIN_EXPIRED`
- `ERROR`

## 6. 推荐接入顺序

建议按下面顺序开发，不要一上来就写完整采集。

### 第一步：只打通首页打开

目标：

- 浏览器能打开平台首页。
- `open_session` 能正常返回 `PlatformSession`。

### 第二步：打通人工登录确认

目标：

- 能识别是否登录成功。
- `check_ready` 能返回准确状态。

建议：

- 优先根据 URL、页面关键文案、搜索框是否存在做判断。
- 不要只依赖单一选择器。

### 第三步：打通搜索链路

目标：

- 输入小区名。
- 提交搜索。
- 到达结果页。

### 第四步：打通筛选链路

目标：

- 正确点击面积筛选。
- 处理该平台的面积档位映射。

说明：

- 面积档位一般属于平台固有规则，放到 `app/platforms/<platform>_constants.py`。
- 不建议放到 `config.py`。

### 第五步：打通在售单价抓取

目标：

- 只抓主结果区。
- 过滤广告、猜你喜欢、推荐位。
- 能处理分页。

说明：

- 分页尽量走真实点击，不建议拼接 URL。
- 如果需要模拟真人，建议保留适当停留时间和滚动动作。

### 第六步：打通详情页与成交抓取

目标：

- 打开详情页。
- 抓小区均价。
- 抓成交记录。
- 按面积筛选成交价。

### 第七步：接入日志与演示脚本

目标：

- 平台结果能出现在服务日志里。
- 单次演示脚本和批量演示脚本能正常展示。

## 7. 推荐实现模板

### `app/platforms/ajk.py`

```python
from app import ajk_adapter
from app.models import InquiryRequest, PlatformSession
from app.platforms.base import PlatformAdapter
from app.platforms.ajk_constants import START_URL


class AjkPlatformAdapter(PlatformAdapter):
    code = "ajk"
    name = "安居客"
    start_url = START_URL

    async def open_session(self, browser) -> PlatformSession:
        page = await browser.get(self.start_url)
        await page
        return PlatformSession(
            code=self.code,
            name=self.name,
            start_url=self.start_url,
            page=page,
            ready=True,
        )

    async def collect(self, browser, session: PlatformSession, request: InquiryRequest):
        return await ajk_adapter.collect(
            browser=browser,
            session=session,
            community_name=request.community_name,
            area_min=request.area_min,
            area_max=request.area_max,
            request_id=request.request_id,
        )

    async def check_ready(self, session: PlatformSession) -> tuple[bool, str]:
        return await ajk_adapter.probe_ready(session.page)
```

### `app/platforms/ajk_constants.py`

```python
START_URL = "https://example.com/"

    (70, 90, "a2"),
    (90, 110, "a3"),

```

## 8. 注册新平台

新增平台文件后，需要改三处。

### 1. 平台导出

修改 `app/platforms/__init__.py`：

```python
from app.platforms.ajk import AjkPlatformAdapter

__all__ = ["PlatformAdapter", "KePlatformAdapter", "AjkPlatformAdapter"]
```

### 2. 平台注册

修改 `app/registry.py`：

```python
from app.platforms import AjkPlatformAdapter, KePlatformAdapter, PlatformAdapter


def build_default_adapters() -> list[PlatformAdapter]:
    return [KePlatformAdapter(), AjkPlatformAdapter()]
```

### 3. 如需平台专属测试脚本，再补脚本

建议命名：

- `app/scripts/ajk_flow_test.py`
- `app/scripts/ajk_quote_detail_test.py`

## 9. 登录与风控处理建议

多平台接入时，最容易出问题的是登录态和风控，不是采集本身。

建议：

- `check_ready` 同时检查 URL、页面关键文案、关键输入框。
- 命中登录页或验证页时，明确返回：
  - `WAIT_MANUAL_VERIFY`
  - `LOGIN_EXPIRED`
- 不要把登录失败误报成 `NO_DATA`。
- 命中人工验证时，尽量保留现场，方便人工接管。

## 10. 调试建议

新平台初次接入时，建议始终开 `--debug`：

```bash
python -m app.scripts.api_server --debug
```

重点观察：

- 搜索前 HTML
- 搜索后 HTML
- 筛选后 HTML
- 分页后 HTML
- 详情页 HTML
- 登录失效或验证跳转现场 HTML

## 11. 接入验收清单

新平台至少要通过以下验收：

- 能打开首页并常驻。
- 能人工登录并确认就绪。
- 能搜索目标小区。
- 能正确点击面积筛选。
- 能抓到主结果区在售单价。
- 能过滤广告和猜你喜欢。
- 能处理分页。
- 能打开详情页。
- 能抓到小区均价。
- 能抓到成交案例并做面积筛选。
- 能返回标准 `PlatformResult`。
- 能在日志里打印房源摘要与均价结果。

## 12. 当前多平台融合策略说明

当前项目虽然支持多平台结构，但最终结果策略仍然比较保守：

- 优先取第一个 `SUCCESS` 平台结果。
- 如果没有成功结果，取第一个非 `ERROR` 结果。

这意味着：

- 现在的“多平台支持”已经具备工程接入能力。
- 当前“所有网站”应理解为“所有已接入平台”。
- 但“多平台结果融合算法”还没有展开设计。

后续接入第二个平台后，再根据真实业务决定是否升级为：

- 多平台取最低值
- 多平台加权平均
- 按城市或平台可信度分层
- 按平台优先级兜底

## 13. 建议的开发节奏

最稳的方式是：

1. 先复制贝壳适配器骨架。
2. 先打通首页、登录、搜索，不急着抓详情。
3. 先做单平台测试脚本。
4. 单平台链路稳定后，再挂到 `registry.py`。
5. 第二个平台稳定后，再讨论多平台融合策略。

这样风险最小，也最符合当前项目阶段。
