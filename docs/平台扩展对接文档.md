## 1. 对接目标

新增一个平台时，需要解决四件事：

1. 打开该平台的常驻首页。
2. 支持人工登录后确认就绪。
3. 接收询价请求并完成采集。
4. 把结果转换成统一的 `PlatformResult`，交给服务层继续计算最终值。

## 2. 现有扩展点

当前项目已经预留的插槽如下：

- `app/platforms/base.py`
  统一平台适配器接口（含 `check_ready` / `keepalive` 基类实现）。
- `app/platforms/ke.py`
  贝壳平台薄壳适配器，可作为新平台模板。
- `app/platforms/ke_constants.py`
  平台固有常量的放置方式示例。
- `app/platforms/adapters/ke.py`
  平台真实采集逻辑示例。
- `app/registry.py`
  平台注册入口。
- `app/runtime.py`
  负责平台会话、保活、任务执行、结果回调。
- `app/service.py`
  汇总平台结果并计算 `quoteAvg`、`dealAvg`、`finalPrice`。

说明：

- 业务目标是面向“当前所有已接入平台”（已接入贝壳、安居客、链家、房天下、乐有家 5 个）。
- 服务层把所有 `SUCCESS` 平台的在售均价、成交单价**跨平台累加平均**后，再走 `decide()` 算最终价（不再是取第一个 `SUCCESS` 平台）。

## 3. 新平台建议文件结构

以平台代号 `ajk` 为例，建议新增这些文件：

```text
app/
├─ parsers/
│  └─ ajk.py               # HTML 解析（纯函数，从页面提取数据）
└─ platforms/
   ├─ ajk.py                 # 薄壳适配器，接入统一接口
   ├─ ajk_constants.py        # 平台固有常量
   └─ adapters/
      └─ ajk.py               # 真实采集逻辑（浏览器操作，解析调 parsers）
```

职责划分：

- `app/parsers/ajk.py`
  HTML 解析（纯函数，无浏览器依赖），从结果页/成交页提取在售快照、成交记录、均价等。
  可独立单测（喂构造 HTML 片段），跟随平台页面结构变化而改。
- `app/platforms/ajk.py`
  薄壳适配器，负责接入统一接口，委托给 adapter。
- `app/platforms/ajk_constants.py`
  平台固有常量，例如首页地址、面积档位、固定选择器。
- `app/platforms/adapters/ajk.py`
  真实采集逻辑，负责页面交互、点击、翻页、抓取、详情页处理。
  解析时 `from app.parsers import ajk as parsers` 调用，不自己写正则。
  以及平台自己的 `probe_ready` / `keepalive` / `detect_block`。

## 4. 必须实现的接口

新平台适配器必须继承 `PlatformAdapter`，并实现以下方法。

> 下文标注 **[基类已实现]** 的表示 `base.py` 已经提供默认实现，子类一般不用重写；
> 只需实现 **[子类实现]** 的部分。

### `open_session` **[子类实现]**

作用：

- 打开平台首页。
- 返回 `PlatformSession`。

要求：

- `code` 必须唯一，例如 `ke`、`ajk`、`lj`、`fang`、`lyj`。
- `name` 为日志和展示使用的平台名称。
- `start_url` 为该平台的常驻首页。
- `page` 为 nodriver 当前页面对象。
- 签名 `open_session(self, browser, new_tab=False)`：多浏览器模式下每个平台
  独占一个浏览器，`new_tab` 传 `False`（导航空白标签页即可），不需要开新标签。

### `collect` **[子类实现]**

作用：

- 执行一次完整的单平台询价采集。

输入：

- `browser`
- `session`
- `request`（含 `city`、`community_name`、`area`、`request_id`）

输出：

- `PlatformResult`

约定：

- 薄壳 `collect()` 开头先调 `check_city_support(request.city)` 检查城市支持，
  不支持则跳过采集、只做保活刷新、返回 `NO_DATA`。
- 城市支持后调 `ensure_city_navigated(session, request.city)` 确保浏览器在目标城市首页
  （城市不同时自动导航，相同则跳过）。
- 然后调 `adapter.collect(..., city=request.city)` 执行真实采集。
- 采集结束前把页面**复位回首页**（调用各 adapter 的 `reset_to_start_page(page, city)`），
  让平台回到待命态，方便下一单和保活。

### `check_ready` **[基类已实现]**

作用：

- 检查当前平台是否已登录、未被风控、且能接单。

基类已实现完整流程，子类**不需要重写**，它内部按顺序做：

1. 取 `body` + 页面 HTML（含最多 2 次 WebSocket 断连重试）。
2. 登录检测（仅当 `requires_login=True`）：正向查"退出"标识 + 反向查"登录"链接兜底。
3. 调用子类的 `_probe_ready` 做平台特有检测（搜索框、筛选区等是否存在）。

### `_probe_ready` **[子类实现]**

作用：

- 基类 `check_ready` 通过登录检测后，交棒给子类做平台特有的就绪检测。

签名：`_probe_ready(self, page, html) -> (bool, str)`

返回：

- `(True, "READY")`
- 或 `(False, "原因描述")`

各平台通常检查搜索框 / 筛选区 / 房源列表是否存在。基类有默认实现直接返回 `(True, "READY")`，
强烈建议子类覆盖，给出更准确的就绪判断。

### `requires_login` **[类属性，可选]**

作用：

- 标记该平台是否需要人工前置登录。

默认 `True`。设为 `False`（如安居客）时，基类 `check_ready` 跳过登录检测，
只走 `_probe_ready`。

### `detect_block` **[子类实现]**

作用：

- 检测当前页面是否被风控 / 登录拦截。

实现分两层：

- 薄壳适配器（`platforms/<code>.py`）里是**实例方法** `detect_block(self, url, html) -> (bool, str)`，
  仅做委托：`return <code>_adapter.detect_block(url, html)`。
- adapter（`platforms/adapters/<code>.py`）里是**模块级函数** `detect_block(url, html) -> (bool, str)`，
  写真正的检测逻辑（验证码/登录 markers、ucid 等）。新平台只需在 adapter 里实现这个模块级函数，
  薄壳照抄委托即可。

返回：

- `(True, "拦截原因")`
- 或 `(False, "")`

各平台在采集关键步骤（搜索后、筛选后、翻页后）通过 `wait_and_reload_after_block`（见下方模块函数表）
统一调用本平台的 `detect_block`，命中则阻塞等人回车后重取。

### `check_city_support` **[基类已实现]**

作用：

- 检查平台是否支持请求中的城市。不支持时返回 `NO_DATA` 结果（含支持城市列表），支持时返回 `None`。

基类已实现完整逻辑，子类**不需要重写**。各薄壳 `collect()` 开头调用：

```python
skip = self.check_city_support(request.city, request.request_id)
if skip is not None:
    # 不支持：保活刷新 + 返回 NO_DATA
    ...
    return skip
```

城市映射数据维护在 `app/platforms/city_map.py` 的 `CITY_MAP` 中。

### `ensure_city_navigated` **[基类已实现]**

作用：

- 确保浏览器已导航到目标城市首页。在调 `adapter.collect` 之前调用。

基类已实现完整逻辑，子类**不需要重写**。内部用 `urlparse` 比较当前页面域名和目标城市域名：

- 域名相同（同城）→ 跳过导航，减少不必要的刷新。
- 域名不同（城市切换）→ `page.get(target_url)` 导航到目标城市首页，更新 `session.page`。

各薄壳 `collect()` 中城市检查通过后调用：

```python
await self.ensure_city_navigated(session, request.city)
```

### `keepalive` **[基类已实现，可覆盖]**

作用：

- 执行轻量保活。

基类默认实现直接复用 `check_ready`。但当前各平台 adapter 都重写了自己的 `keepalive`/`probe_ready`：
先探测，就绪则滚动到顶部；未就绪则刷新回首页再探测一次。新平台可参照同款写法。

### 模块级公共函数（`base.py` 提供，各平台共用）

这些是在实际对接过程中沉淀下来的通用能力，直接从 `base` 导入复用：

| 函数 | 用途 |
|------|------|
| `human_linger(page, page_no, linger_seconds=None)` | 翻页后模拟真人停留，间隔取 `config.PAGE_LINGER_SECONDS` |
| `wait_for_manual_unblock()` | 命中风控/登录拦截时，终端提示并阻塞等回车 |
| `wait_and_reload_after_block(tab, detect_func, label)` | 详情/成交/翻页被风控时的**统一处理入口**：取页面 → `detect_func` 精确检测 → 未命中再叠通用兜底 `is_generic_captcha_page` → 命中则 `wait_for_manual_unblock` 等人回车 → 重取，最多 2 次。各 adapter 在打开详情/成交 tab、翻页后调用本函数，替代各自手写的「detect_block → 等人 → 重取」 |
| `is_generic_captcha_page(html)` | 通用风控兜底：判断 HTML 是否为验证码拦截页（跨平台共性，判据见 `base.py`）。只在各平台精确 `detect_block` 都未命中时作为最后一道网，防止某平台验证码 markers 被删空后整体漏判 |
| `short_circuit_result(name, status, reason, request_id, started_at, detail_url=None)` | 构造三类状态短路返回（`NO_DATA`/`WAIT_MANUAL_VERIFY`/`LOGIN_EXPIRED` 等）的工厂，统一计算 `elapsed_seconds`。各 adapter 的短路返回改调本函数，避免 5-6 行模板重复 |
| `_human_click(page, element, label)` | 真人节奏点击元素，优先 JS click，失败降级 mouse_click，随机间隔 |
| `safe_select_and_click(page, selector, *, dump_fn, dump_name, detect_fn, block_label, click_label)` | 通用的"安全选择+点击"：找不到元素时 dump 现场 + 风控检测 + 恢复后重试 + 点击。封装翻页时「select → 找不到 → dump → detect_block → wait_and_reload → retry → 点击」的完整逻辑，返回 Element 或 None（优雅停止信号）。各平台翻页按钮点击统一调本函数 |
| `check_empty_listing_page(page_no, page_count, consecutive_empty, total_pages, platform)` | 翻页空页检测（4 个翻页平台共用）：首页空但 total_pages>0 → `log.error` + 立即停止（total_pages 是旧计数，实际无结果）；连续空页 ≥ 2 → `log.warning` + 停止；单页空 → `log.warning`。返回 `(should_stop, updated_consecutive_empty)`，各 adapter 在 `_collect_listing_pages` 循环内解析完页面数据后调用 |
| `click_area_segment(page, area, parse_func, platform_code)` | 动态读取页面面积档位并点击匹配项（所有平台共用），各平台只需传入自己的 `parsers.parse_area_segments` |
| `community_name_match(request_name, captured_name)` | 比较请求小区名与抓取快照的结构化小区名；不允许整页 HTML、房源标题或搜索词参与匹配 |
| `has_matching_community_snapshots(snapshots, community_name)` | 判断搜索结果里是否至少存在一条命中目标小区的快照，用于搜索后的首轮校验 |
| `filter_snapshots_by_community(snapshots, community_name)` | 只按 `ListingSnapshot.community_name` 过滤抓取数据，供逐页过滤和返回前防御校验共用 |
| `prepare_listing_data(snapshots, community_name)` | 过滤目标小区快照，并从同一批快照同步生成 `quote_prices`，避免明细与均价数据源不一致 |

## 5. `PlatformResult` 要求

平台采集最终必须返回统一结构：

- `name`
- `status`
- `community_avg_price`
- `quote_prices`
- `deal_prices`
- `deal_records`
- `deal_source`
- `reason`
- `request_id`
- `detail_url`
- `elapsed_seconds`
- `listing_snapshots`

字段含义：

- `community_avg_price`
  平台详情页提供的小区参考均价。当前服务层会优先把它当作 `quoteAvg` 候选。
- `quote_prices`
  目标小区在售单价列表，必须从过滤后的 `listing_snapshots` 同步生成，不能从过滤前数据单独计算。
- `deal_prices`
  成交单价列表，注意这里应该是面积筛选后的结果。
- `deal_records`
  成交记录明细（用于日志打印展示），元素为 `{area, date, price}` 之类的 dict。
- `deal_source`
  成交来源说明，便于追溯：`"成交记录"` / `"挂牌均价顶替"` / `"小区均价顶替"` / `"无"`。
  无成交记录的平台（安居客、乐有家）会在此标注是哪种均价顶替了 `deal_prices`。
- `listing_snapshots`
  目标小区房源摘要，用于日志打印、演示展示和后续入库；不得包含宽搜索混入的无关小区房源。

`status` 当前建议使用以下值：

- `SUCCESS`
- `NO_DATA`
- `WAIT_MANUAL_VERIFY`
- `LOGIN_EXPIRED`
- `ERROR`

## 6. 推荐接入顺序

建议按下面顺序开发，不要一上来就写完整采集。

### 第一步：只打通首页打开

目标：

- 浏览器能打开平台首页。
- `open_session` 能正常返回 `PlatformSession`。

### 第二步：打通人工登录确认

目标：

- 能识别是否登录成功。
- `check_ready` 能返回准确状态。

建议：

- 优先根据 URL、页面关键文案、搜索框是否存在做判断。
- 不要只依赖单一选择器。

### 第三步：打通搜索链路

目标：

- 输入小区名。
- 提交搜索。
- 到达结果页。

说明：

- 搜索完成后先拿结果页 HTML 跑 `parsers.parse_listing_snapshots(keyword_html)`，得到已经结构化的抓取数据 `keyword_snaps`。
- 立刻调用 `has_matching_community_snapshots(keyword_snaps, request.community_name)` 做首轮校验；若没有任何匹配小区，直接返回 `NO_DATA`，不要继续做面积筛选。
- 命中判断只能比较 `request.community_name` 与每条 `ListingSnapshot.community_name`；禁止用整页 HTML、房源标题、搜索框值或搜索词命中情况代替抓取数据字段。
- 底层统一走 `community_name_match(...)`，容忍分期、空白、标点和明确的简称/前缀差异，但不得用公共片段相似度猜测两个小区相同。

### 第四步：打通筛选链路

目标：

- 正确点击面积筛选。

说明：

- 面积筛选已统一为**动态读取页面档位 + 点击对应链接**的方式。
  流程由 `base.py` 的 `click_area_segment(page, area, parse_func, platform_code)` 统一实现，
  各平台只需在自己的 parser 中实现 `parse_area_segments(html)` 函数。
- `parse_area_segments` 从页面 HTML 中实时解析面积档位列表，
  返回格式 `[(text, min, max), ...]`，如 `[("50㎡以下", 0, 50), ("70-90㎡", 70, 90), ...]`。
- 档位因城市而异，不能硬编码，必须从 HTML 实时读取。
- `click_area_segment` 内部做左闭右开匹配（`area >= min 且 area < max`），
  超出最大档位自动取末档；档位都是 `<a>` 链接，点击即筛选，不需要填输入框或点确定。

在 adapter 的 `_do_collect` 中调用方式：

```python
from app.platforms.base import click_area_segment

# parser 已 import 为 parsers（如 from app.parsers import ajk as parsers）
await click_area_segment(main_page, area, parsers.parse_area_segments, "ajk")
```

### 第五步：打通在售单价抓取

目标：

- 只抓主结果区。
- 过滤广告、猜你喜欢、推荐位。
- 能处理分页。

说明：

- 分页尽量走真实点击，不建议拼接 URL。
- 如果需要模拟真人，建议保留适当停留时间和滚动动作。
- 每抓完一页就把页面 HTML 解析成 `page_snapshots`，立即调用 `filter_snapshots_by_community(page_snapshots, request.community_name)`；只累计匹配目标小区的快照，不允许无关小区进入后续入库数据。
- 第 1 页非空但全部不匹配目标小区时，立即停止该平台采集并返回 `NO_DATA`，不再打开详情或抓取后续页面。
- 第 2 页及以后，非空页面若全部不匹配目标小区，应丢弃该页并立即停止后续翻页；此前已抓到的目标小区数据继续参与结果。混合页面只保留匹配快照并继续翻页。
- 空页不是“无关小区页”，仍调用 `base.py` 的 `check_empty_listing_page` 走既有空页规则：首页空立即停止，连续 2 页空停止。
- 返回 `SUCCESS` 前再调用 `prepare_listing_data(...)` 做一次防御校验，并从过滤后的同一批快照生成 `quote_prices`，确保房源明细和在售价格完全一致；过滤后为空应返回 `NO_DATA`。
- 这套过滤对有分页和无分页平台都适用。像安居客这种滚动到底的单页平台，也要在滚动完成后调用 `prepare_listing_data(...)`。
- 翻页时调用 `base.py` 的 `check_empty_listing_page` 做空页检测：首页空但 total_pages>0
  会立即 `log.error` 并停止（total_pages 可能是旧计数，实际无结果）；连续 2 页空也会
  `log.warning` 并停止。避免程序静默翻 N 页空数据且无任何报错。

### 第六步：打通详情页与成交抓取

目标：

- 打开详情页。
- 抓小区均价。
- 抓成交记录。
- 按面积筛选成交价。

### 第七步：接入日志与演示脚本

目标：

- 平台结果能出现在服务日志里。
- 单平台 MVP 脚本能跑通完整链路并展示结果。

### `app/platforms/ajk.py`

```python
from app.platforms.adapters import ajk as ajk_adapter
from app.core.models import InquiryRequest, PlatformSession
from app.platforms.base import PlatformAdapter
from app.platforms.ajk_constants import START_URL

import logging

log = logging.getLogger(__name__)


class AjkPlatformAdapter(PlatformAdapter):
    code = "ajk"
    name = "安居客"
    start_url = START_URL
    requires_login = False  # 安居客无需登录；需要登录的平台省略此行（默认 True）

    async def open_session(self, browser, new_tab=False) -> PlatformSession:
        page = await browser.get(self.start_url, new_tab=new_tab)
        await page
        return PlatformSession(
            code=self.code,
            name=self.name,
            start_url=self.start_url,
            page=page,
            ready=True,
        )

    async def collect(self, browser, session: PlatformSession, request: InquiryRequest):
        # 城市支持检查：不支持则跳过询价，只做保活刷新
        skip = self.check_city_support(request.city, request.request_id)
        if skip is not None:
            try:
                session.page = await session.page.get(self.start_url)
                await session.page
            except Exception as exc:
                log.warning("failed to keepalive %s page: %s", self.code, exc)
            return skip

        # 确保浏览器在目标城市首页（城市不同时先导航过去）
        await self.ensure_city_navigated(session, request.city)

        result = await ajk_adapter.collect(
            browser=browser,
            main_page=session.page,
            community_name=request.community_name,
            area=request.area,        # 精确面积，用于动态档位筛选
            request_id=request.request_id,
            city=request.city,         # 城市，用于动态获取该城市首页 URL
        )
        # 采集结束复位回首页（同城市），回到待命态
        try:
            session.page = await ajk_adapter.reset_to_start_page(session.page, request.city)
        except Exception:
            pass
        return result

    async def _probe_ready(self, page, html: str) -> tuple[bool, str]:
        """登录检测由基类 check_ready 负责，这里只做平台特有检测。"""
        return await ajk_adapter.probe_ready(page)

    def detect_block(self, url: str, html: str) -> tuple[bool, str]:
        return ajk_adapter.detect_block(url, html)
```

## 9. 登录与风控处理建议

多平台接入时，最容易出问题的是登录态和风控，不是采集本身。

建议：

- `check_ready` 同时检查 URL、页面关键文案、关键输入框。
- 命中登录页或验证页时，明确返回：
  - `WAIT_MANUAL_VERIFY`
  - `LOGIN_EXPIRED`
- 不要把登录失败误报成 `NO_DATA`。
- 命中人工验证时，尽量保留现场，方便人工接管。

登录失效的两种形态（`detect_block` 应都覆盖）：

- **硬失效**：跳转到登录页，URL 含 `login`/`passport`/`signin`。各平台 `detect_block` 的 URL 检测覆盖。
- **软失效**：URL 没变（仍停在当前业务页），但登录掉了（弹窗形式）。
  贝壳/链家共用 `ljConf` 框架，可用后端注入的 `ucid` 判定：未登录为空字符串 `ucid:''`、
  已登录为数字。判据见 `adapters/ke.py`/`adapters/lj.py` 的 `_is_login_expired_html`
  （正则 `ucid\s*:\s*''\s*,?\s*cdn`，用"紧跟 cdn"锚定 ljConf 避免误匹配无关 ucid）。
  其它平台无对照样本时暂只用 URL 检测，拿到"未登录"对照 dump 后再补 HTML/ucid 层。

风控处理统一走 `wait_and_reload_after_block`（见 §4 模块函数表）：

- 各 adapter 在打开详情/成交 tab、翻页后调用本函数，不要各自手写「detect → 等人 → 重取」。
- 它内部先用本平台精确 `detect_block`，未命中再叠通用兜底 `is_generic_captcha_page`，
  保证某平台验证码 markers 被删空后仍能兜住验证码页（防整体漏判）。
- 命中则 `wait_for_manual_unblock()` 阻塞等终端回车（人去浏览器处理），回车后重取，最多 2 次。
- 验证码/登录的精确 markers 各平台维护在 adapter 内（风控页是各平台自有安全系统，特征不通用），
  改 markers 时务必基于真实 dump 样本（`--debug` 导出），不要盲写——这是项目反复踩过的坑。

## 12. 当前多平台融合策略说明

当前结果策略（已实现，见 `app/core/algorithm.py` 的算法策略接口和注册表）：

- `AlgorithmInput` 是算法层的统一输入结构，服务层只负责从 `PlatformResult` 组装它。
- `AlgorithmStrategy` 是算法策略接口，算法实现通过注册表按 `algorithmMode` 选择。
- 新增算法只需实现策略接口并加入注册表，不需要修改平台适配器或采集流程。

### `default`

- 每个 `SUCCESS` 平台先得到一个在售代表均价：优先使用 `community_avg_price`，否则使用该平台 `quote_prices` 的均值。
- 各平台代表均价再做平台等权平均，作为 `quoteAvg`。
- 所有 `SUCCESS` 平台的 `deal_prices` 合并后取均值，作为 `dealAvg`。
- 最后把 `quoteAvg` / `dealAvg` 交给 `decide()`，保持成交+在售综合决策规则不变。

### `quote_only`

- 忽略 `community_avg_price` 和 `deal_prices`。
- 把所有 `SUCCESS` 平台的 `quote_prices` 合并成一个列表，再统一求均值，作为 `quoteAvg`。
- 最后把 `quoteAvg` 交给 `decide_quote_only()`，使用 `quoteOnlyDiscount` 打折输出。
- 同一房源若同时出现在多个平台，按多个平台的抓取记录分别计入；跨平台去重不属于当前算法规则。

空数据的平台不参与计算。两种模式的聚合和决策都由算法层统一处理，服务层只负责传入标准化的 `PlatformResult` 数据。

后续若业务需要更精细的策略，再考虑升级为：

- 多平台取最低值
- 多平台加权平均
- 按城市或平台可信度分层
- 按平台优先级兜底

### 算法模式

系统支持两套算法，通过 API 入参 `algorithmMode` 切换：

| 模式 | 参数值 | 算法 | 说明 |
|------|--------|------|------|
| 默认 | `"default"` | `decide(quote_avg, deal_avg)` | 成交+在售综合决策（4 条分支） |
| 纯在售 | `"quote_only"` | `decide_quote_only(quote_avg)` | 只看在售均价，打折扣输出 |

- `algorithmMode` 为可选字段，不传默认 `"default"`。
- 两套算法对各平台采集流程无影响 — adapter 不作任何改动，服务层只调用算法策略注册表。
- 纯在售折扣系数 `quoteOnlyDiscount` 默认 0.9，可通过 `PUT /admin/algorithm/quote-only-discount` 动态调整。

## 13. 建议的开发节奏

最稳的方式是：

1. 先复制贝壳适配器骨架。
2. 先打通首页、登录、搜索，不急着抓详情。
3. 先做单平台测试脚本。
4. 单平台链路稳定后，再挂到 `registry.py`。
5. 第二个平台稳定后，再讨论多平台融合策略。

这样风险最小，也最符合当前项目阶段。

## 14. 多城市支持

### 概述

系统支持广东省 21 个地级市的多城市询价。API 入参 `city` 为必填字段（城市、小区、面积三要素）。
各平台 URL 前缀命名规则不统一（缩写/全拼混用），不能用规则推导，必须维护显式映射表。

### 城市映射表

映射数据维护在 `app/platforms/city_map.py`：

```python
from app.platforms.city_map import get_start_url, is_city_supported, get_city_prefix

# 获取某平台某城市的起始 URL
url = get_start_url("ke", "深圳")       # → "https://sz.ke.com/ershoufang/"

# 检查平台是否支持某城市
is_city_supported("ajk", "汕头")         # → True

# 获取城市前缀（lyj 搜索 URL 需要用）
prefix = get_city_prefix("lyj", "广州")  # → "guangzhou"
```

### 各平台城市覆盖

| 平台 | 覆盖数 | 不支持的城市 |
|------|--------|-------------|
| 安居客(ajk) | 21/21 | 无 |
| 房天下(fang) | 21/21 | 无 |
| 贝壳(ke) | 12/21 | 汕头、韶关、茂名、梅州、汕尾、河源、潮州、揭阳、云浮 |
| 链家(lj) | 10/21 | 汕头、韶关、茂名、梅州、汕尾、河源、阳江、肇庆、潮州、揭阳、云浮 |
| 乐有家(lyj) | 9/21 | 汕头、韶关、湛江、肇庆、茂名、梅州、汕尾、河源、阳江、潮州、揭阳、云浮 |

5 平台全部覆盖的城市（9 个）：广州、深圳、珠海、佛山、东莞、中山、惠州、江门、清远。

### 采集流程中的城市处理

薄壳 `collect()` 按以下顺序处理城市：

1. **`check_city_support(city)`** — 不支持则打印日志 + 保活刷新 + 返回 `NO_DATA`
2. **`ensure_city_navigated(session, city)`** — 城市不同时导航到目标城市首页
3. **`adapter.collect(..., city=city)`** — 传入 city 给 adapter
4. **`reset_to_start_page(page, city)`** — 采集结束复位到同城市首页

adapter 内部通过 `get_start_url(code, city)` 动态获取 URL，不再使用硬编码的 `START_URL`。
`START_URL` 仅保留用于 `open_session` 初始打开和城市不支持时的保活刷新。
